# 要件定義

## トリガー

「要件定義したい」「要件を決めたい」「仕様を整理したい」「機能を決めたい」等

## 概要

クライアントとの対話を通じて、プロジェクトの要件を整理し、`requirements.md`（要件定義書）を生成する。
ヒアリングは段階的に進め、クライアントを圧倒しないよう注意する。

## 手順

### Step 1: 目的と背景の確認

TASKLIST.md が存在する場合、そこに書かれたプロジェクト概要を読み、確認から始める:

「タスクリストによると、[プロジェクト概要] ですね。ここからもう少し詳しく聞いていきます。」

TASKLIST.md がない場合は、ゼロからヒアリングする:

1. 「このプロジェクトで一番実現したいことは何ですか？」
2. 「なぜこれが必要なのですか？今どんな困りごとがありますか？」

### Step 2: ターゲットユーザーの明確化

1. 「これを使うのは誰ですか？（例: 社員、お客さん、自分だけ、など）」
2. 「その人たちは普段どんなツールを使っていますか？ITに詳しい方ですか？」
3. 「ユーザーは何人くらいを想定していますか？」

### Step 3: 機能要件のヒアリング

段階的に聞く。一度に全部聞かない。

**まず大枠:**
「このサービス/ツールで、ユーザーが『できること』を思いつく限り挙げてください。」

**回答を受けて整理:**
挙がった機能を「必須」と「あると嬉しい」に分類して提示する:

「整理するとこんな感じですか？」

```
■ 必須（これがないと成り立たない）
- 機能A
- 機能B

■ あると嬉しい（後から追加でもOK）
- 機能C
- 機能D
```

「追加や修正があれば教えてください。」

**各機能を深掘り（必須機能のみ）:**
必須機能それぞれについて:
- 「[機能名] について、もう少し詳しく教えてください。具体的にどういう動きをイメージしていますか？」

### Step 4: 非機能要件の確認

技術的な言葉を避けて聞く:

1. **対応デバイス**: 「スマホから使えた方がいいですか？パソコンだけでOKですか？」
2. **速度**: 「たくさんの人が同時に使う場面はありますか？」
3. **セキュリティ**: 「扱うデータに個人情報やお金に関するものは含まれますか？」
4. **既存システム連携**: 「今使っているシステムやツール（スプレッドシート、Slackなど）と連携させたいものはありますか？」

すべて聞く必要はない。プロジェクトに関係なさそうなものはスキップする。

### Step 5: 制約条件の確認

1. **期限**: 「いつまでに使えるようにしたいですか？」
2. **予算**: 「予算感はありますか？（なければ大丈夫です）」
3. **技術制約**: 「使いたいサービスや、逆に使いたくないものはありますか？」

### Step 6: 要件定義書の生成

ヒアリング内容をもとに `requirements.md` を生成する。
生成前にチャットで内容をプレビューし、クライアントに確認する。

**ファイル形式:**

```markdown
# 要件定義書: [プロジェクト名]

作成日: [YYYY-MM-DD]

## 1. プロジェクト概要

### 目的
[1〜2文で]

### 背景・課題
[現状の課題や、このプロジェクトが必要な理由]

## 2. ターゲットユーザー

| 項目 | 内容 |
|---|---|
| 主なユーザー | [誰が使うか] |
| ITリテラシー | [高/中/低] |
| 想定ユーザー数 | [人数] |
| 利用デバイス | [PC/スマホ/両方] |

## 3. 機能要件

### 必須機能

| # | 機能名 | 概要 |
|---|---|---|
| 1 | [機能名] | [何ができるか] |
| 2 | [機能名] | [何ができるか] |

### あると嬉しい機能

| # | 機能名 | 概要 |
|---|---|---|
| 1 | [機能名] | [何ができるか] |

## 4. 非機能要件

| 項目 | 内容 |
|---|---|
| 対応デバイス | [PC/スマホ/両方] |
| セキュリティ | [必要な対策] |
| 外部連携 | [連携先サービス] |

## 5. 制約条件

| 項目 | 内容 |
|---|---|
| 期限 | [いつまでに] |
| 予算 | [予算感] |
| 技術制約 | [あれば] |

## 6. 今後の進め方

[次のステップの提案]
```

### Step 7: 確認と保存

1. チャットに要件定義書の内容をプレビュー表示する
2. 「この内容で問題なければ、要件定義書として保存します。修正したい箇所があれば教えてください。」
3. クライアントが OK を出したら、ワークスペースルートに `requirements.md` を保存する
4. 「要件定義書を作成しました！」と伝える

## 注意事項

- ヒアリングは対話形式で進める。フォームのように一問一答にしない
- クライアントが「わからない」と言ったら、具体例を出して助ける
- 完璧を求めない。「後から変更できます」と伝えて安心させる
- 技術的な判断（フレームワーク選定など）はこのスキルでは行わない。要件として記録するだけ
- 要件定義書は開発者（佐藤や別のエンジニア）が読んで開発に着手できるレベルを目指す
