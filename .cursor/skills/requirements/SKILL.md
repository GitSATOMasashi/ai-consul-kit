# 要件定義

## トリガー

「要件定義したい」「要件を決めたい」「仕様を整理したい」「機能を決めたい」「技術要件を詰めたい」等

## 概要

`ux-design.md`（理想体験シナリオ）を入力として、その体験を実現するための技術要件を対話で詰めていく。
ゴールは **Cursor のエージェントモードに渡したら、判断に迷うことなく実装を完了できるレベルの `requirements.md`** を作ること。
エージェントに判断の余地を残さない。すべての振る舞い・データ・エラー対応を明記する。

## 進め方の原則

- ux-design.md を必ず最初に読む。理想体験を技術要件に翻訳するのがこのスキルの仕事
- クライアントに技術的な判断を求めすぎない。選択肢を 2〜3 個に絞って提示し、メリデメを説明して選んでもらう
- 「これでエージェントが迷わず作れるか？」を常に自問する
- 曖昧なまま残さない。「〜など」「〜等」「適宜」は禁止。すべて具体的に書く

## 手順

### Step 1: ux-design.md を読み込む

ワークスペースルートの `ux-design.md` を読む。

**存在する場合:**
理想体験シナリオを要約して確認する:

「UX設計書によると、こういう体験を目指していますね:
- [シナリオ1の要約]
- [シナリオ2の要約]
- 避けたいこと: [要約]

この体験を実現するために、技術的な部分を一つずつ決めていきましょう。」

**存在しない場合:**
「要件定義の前に、まず理想のユーザー体験を整理した方がいいかもしれません。先にユーザー体験の具体化をやりますか？」
クライアントが要件定義から始めたい場合は、Step 2 でユーザー体験も簡易的にヒアリングしながら進める。

### Step 2: 技術スタックの決定

理想体験をもとに、使う技術・サービスを決めていく。
クライアントの技術レベルに合わせて説明する。

**聞くこと:**

チャネル/プラットフォーム:
- 「ユーザーとのやりとりは [UXで出たチャネル: LINE/Web/Slack等] でしたね。これで決まりですか？」
- 「管理画面は必要ですか？スプレッドシートで十分ですか？」

データの保存先:
- 「データはどこに保存しますか？選択肢としては:」
  - 「A. Google スプレッドシート — 簡単に見れる、複数人で共有しやすい」
  - 「B. データベース — 大量データに強い、検索が速い」
  - 「C. その他（Hubspot、Notion 等、既に使っているもの）」

外部連携:
- 「今使っているサービスで、連携させたいものはありますか？（Hubspot、Slack、Google カレンダー等）」
- 連携先ごとに: 「どんなデータをやりとりしますか？」

AI の利用:
- 「AI を使う部分はありますか？ある場合、何をさせたいですか？」
- 「AI に任せる範囲と、人間がチェックする範囲を決めましょう」

ホスティング/実行環境:
- 「常時動いている必要がありますか？それとも手動で実行する形でもOKですか？」
- 選択肢を提示: 「A. サーバーで常時稼働」「B. スケジュール実行」「C. 手動実行」

**整理して確認:**

```
■ 技術スタック（案）
- チャネル: LINE Messaging API
- バックエンド: Node.js
- データ保存: Google スプレッドシート
- AI: Claude API（回答生成）
- ホスティング: Render.com
- 外部連携: Hubspot（顧客データ取得）
```

「この組み合わせでいいですか？」

### Step 3: 機能一覧の洗い出し

ux-design.md のシナリオから、必要な機能を漏れなく抽出する。

**やり方:**
シナリオの各ステップを見て、「これを実現するにはどんな機能が必要か？」を考える。

例: 「お客さんが LINE で質問 → AI が自動回答」というシナリオなら:
- LINE メッセージ受信機能
- AI 回答生成機能
- LINE メッセージ送信機能
- 回答できなかった場合のスタッフ通知機能

**クライアントに確認:**
「シナリオから、こういう機能が必要だと思います:」

```
■ 機能一覧
1. [機能名] — [1行で何をするか]
2. [機能名] — [1行で何をするか]
3. [機能名] — [1行で何をするか]
...
```

「漏れている機能や、不要な機能はありますか？」

### Step 4: 各機能の詳細仕様を詰める

機能ごとに、以下を全て決める。1 機能ずつ順番に進める。

**各機能で決めること:**

1. **トリガー**: 何がきっかけでこの機能が動くか
2. **入力**: どんなデータを受け取るか（形式・例を具体的に）
3. **処理**: 何をするか（ステップを箇条書きで）
4. **出力**: 何を返すか / どこに保存するか（形式・例を具体的に）
5. **エラー時**: うまくいかなかったとき、何が起きるか
6. **制約**: レスポンス時間、データ量の上限、同時実行数 等

**クライアントへの聞き方（例: リマインド送信機能）:**

- 「リマインドは何時間前に送りますか？」→ 「24時間前と1時間前の2回」
- 「送る内容は毎回同じですか？テンプレートですか？」→ 「テンプレートで、名前と日時が入る」
- 「送信先はどこから取得しますか？」→ 「スプレッドシートのリスト」
- 「送信に失敗したらどうしますか？」→ 「スタッフに Slack 通知」
- 「1日に最大何件くらい送りますか？」→ 「最大50件」

### Step 5: データ構造の定義

各機能で扱うデータを整理する。

**決めること:**
- どんなデータ項目があるか（カラム名・型・例）
- どこに保存するか
- 誰が読み書きするか

**クライアントへの聞き方:**
- 「リマインド送信に必要な情報を整理しましょう。お客さんの名前、連絡先、リマインドの日時、あと何が必要ですか？」
- 「今スプレッドシートで管理しているデータがあれば、見せてもらえますか？」

**整理して提示:**

```
■ データ: 顧客リスト
保存先: Google スプレッドシート「顧客管理」シート

| カラム名 | 型 | 例 | 説明 |
|---|---|---|---|
| customer_name | テキスト | 田中太郎 | 顧客名 |
| line_user_id | テキスト | U1234abcd | LINE ユーザーID |
| remind_date | 日付 | 2026-04-01 | リマインド対象日 |
| status | テキスト | 未送信 | 未送信/送信済み/失敗 |
```

### Step 6: ファイル構成の設計

プロジェクトのディレクトリ構造を決める。

```
project-name/
├── src/
│   ├── index.js          — エントリーポイント
│   ├── line-handler.js   — LINE メッセージ受信・送信
│   ├── ai-responder.js   — AI 回答生成
│   ├── reminder.js       — リマインド送信
│   └── sheets.js         — スプレッドシート読み書き
├── .env                  — 環境変数（APIキー等）
├── package.json
└── README.md
```

「こういうファイル構成で進めようと思います。わからないところがあれば聞いてください。」

### Step 7: 環境変数・認証情報の整理

必要な API キーや認証情報をリストアップする。

```
■ 必要な環境変数
| 変数名 | 説明 | 取得方法 |
|---|---|---|
| LINE_CHANNEL_SECRET | LINE チャネルシークレット | LINE Developers で取得 |
| LINE_CHANNEL_ACCESS_TOKEN | LINE アクセストークン | LINE Developers で取得 |
| ANTHROPIC_API_KEY | Claude API キー | Anthropic Console で取得 |
| GOOGLE_SHEETS_ID | スプレッドシートID | スプレッドシートのURLから |
```

「これらの設定が必要になります。後で一緒に取得しましょう。」

### Step 8: requirements.md の生成

全てをまとめて `requirements.md` を生成する。
生成前にチャットで内容をプレビューし、クライアントに確認する。

**ファイル形式:**

```markdown
# 要件定義書: [プロジェクト名]

作成日: [YYYY-MM-DD]
ベース: ux-design.md（理想体験シナリオ）

## 1. プロジェクト概要

### 目的
[1〜2文]

### 実現したい体験（ux-design.md より）
[理想体験の要約を箇条書き]

## 2. 技術スタック

| 項目 | 技術/サービス | 選定理由 |
|---|---|---|
| チャネル | [LINE/Web/Slack等] | [なぜこれか] |
| バックエンド | [言語/FW] | [なぜこれか] |
| データ保存 | [DB/スプシ等] | [なぜこれか] |
| AI | [Claude/GPT等] | [なぜこれか] |
| ホスティング | [Render/Vercel等] | [なぜこれか] |
| 外部連携 | [Hubspot/Slack等] | [なぜこれか] |

## 3. 機能一覧

| # | 機能名 | 概要 | 優先度 |
|---|---|---|---|
| 1 | [機能名] | [何をするか] | 必須/後回しOK |

## 4. 機能詳細

### 4.1 [機能名]

**トリガー:** [何がきっかけで動くか]

**入力:**
- [データ項目]: [型] — 例: [具体例]

**処理:**
1. [ステップ1]
2. [ステップ2]
3. [ステップ3]

**出力:**
- [何を返すか/保存するか]: [形式] — 例: [具体例]

**エラー時:**
- [エラー条件] → [何が起きるか]

**制約:**
- レスポンス: [何秒以内]
- データ量: [上限]

### 4.2 [機能名]
[同じ形式で繰り返し]

## 5. データ構造

### 5.1 [データ名]

保存先: [どこに]

| カラム名 | 型 | 例 | 説明 |
|---|---|---|---|
| [名前] | [型] | [例] | [説明] |

## 6. ファイル構成

```
[ディレクトリツリー]
```

各ファイルの役割:
- `[ファイル名]` — [何をするファイルか]

## 7. 環境変数

| 変数名 | 説明 | 取得方法 |
|---|---|---|
| [変数名] | [説明] | [どこで取得するか] |

## 8. 制約条件

| 項目 | 内容 |
|---|---|
| 期限 | [いつまでに] |
| 予算 | [月額ランニングコスト等] |
| セキュリティ | [個人情報の扱い等] |

## 9. 開発の進め方

### フェーズ 1（MVP）
- [ ] [最小限で動くものに必要な機能]
- [ ] [最小限で動くものに必要な機能]

### フェーズ 2（改善）
- [ ] [追加機能]
- [ ] [追加機能]
```

### Step 9: 確認と保存

1. チャットに要件定義書の内容をプレビュー表示する
2. 「この要件定義書があれば、AIエージェントがこのまま開発に着手できます。確認してください。」
3. 特に以下の観点で確認を促す:
   - 「曖昧な部分が残っていないか」
   - 「エージェントが『これどうすればいい？』と迷いそうなところがないか」
   - 「抜けている機能がないか」
4. クライアントが OK を出したら、ワークスペースルートに `requirements.md` を保存する
5. 「要件定義書を作成しました！この内容をもとに開発を始められます。」

## 注意事項

- このスキルのゴールは「エージェントが迷わず開発できる指示書」を作ること
- 「〜など」「〜等」「適宜対応」「必要に応じて」は使わない。すべて具体的に書く
- クライアントが判断できない技術的な部分は、AIが最適な選択肢を提案して理由を説明する
- ux-design.md の理想体験を技術に翻訳するのが仕事。理想体験を勝手に変えない
- 1機能ずつ丁寧に詰める。焦って全体を一気にやらない
- 開発フェーズ分けを必ず入れる。全部一度に作ろうとしない
