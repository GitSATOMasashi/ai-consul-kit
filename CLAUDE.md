# ai-consul-kit

AIコンサル業務の定型作業をCursorスキルとしてパッケージ化したリポジトリ。
クライアントがAIと対話するだけで、プロジェクトの立ち上げから要件定義まで自走できる。

## 対象ユーザー

- 技術レベルはほぼゼロ（PC操作とChatGPTでの対話ができる程度）
- Cursor、Git、ターミナルの経験なし
- Mac 環境

## コミュニケーション方針

このリポジトリを使うクライアントと対話する際は、以下を必ず守ること:

1. **やさしい日本語**で話す。専門用語を使う場合は必ず補足説明をつける
2. **一度に聞く質問は2〜3個まで**。大量の質問で圧倒しない
3. **具体例を添える**。抽象的な説明だけで終わらせない
4. **次にやることを明示する**。クライアントが迷わないようにする
5. **確認を挟む**。長い作業の途中で「ここまで大丈夫ですか？」と聞く

## ディレクトリ構成

```
ai-consul-kit/
├── CLAUDE.md              ← このファイル
├── .cursor/
│   ├── rules/
│   │   ├── task-router.mdc    ← タスクリスト駆動のメインルーター
│   │   └── setup-trigger.mdc  ← 初回セットアップ誘導
│   └── skills/
│       ├── setup-guide/       ← Cursor/Git/GitHub セットアップガイド
│       ├── tasklist/          ← タスクリスト生成（テンプレート付き）
│       └── requirements/      ← 要件定義
├── .gitignore
└── README.md              ← クライアント向け最小限ガイド
```

## 運用モデル

### タスクリスト駆動

すべての進行は `TASKLIST.md`（クライアントのローカルに生成）を軸に進む。

1. クライアントが「始めたい」と話しかける
2. AIがヒアリングし、タスクリストを生成
3. タスクリストの順番に沿って、AIが次のスキルを誘導
4. スキル完了 → タスクリストにチェック → 次へ

### リポジトリ権限

- クライアントには **read-only** アクセスのみ付与（clone/pull のみ）
- 成果物（TASKLIST.md、requirements.md 等）はクライアントのローカルに生成される
- スキル更新時は佐藤が push → クライアントが `git pull` で反映

### スキル発動の仕組み

`.cursor/rules/` のルールファイルが発話パターンにマッチし、対応する `.cursor/skills/` の SKILL.md を読み込んで実行する。

## 設計原則

1. **タスクリストが背骨** — 全ての進行はタスクリストに依存する
2. **スキルは自己完結** — 各スキルは単体で動作し、成果物を出力する
3. **クライアントはチャットするだけ** — ファイル操作やコマンド入力は求めない
4. **段階的に育てる** — 初期は一般的なベストプラクティス、後から佐藤のメソッドを反映
